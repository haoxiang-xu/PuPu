3 种实现方案对比

点击卡片选择,或在终端里告诉我你的想法

01
CONSERVATIVE

Character-Store Clone

复用 character 系统的同形结构,新增一套并行的 agent_store / service / registry / seeding / routes。

✓

  • 最小风险
  • 同形代码好维护
  • 可以先写测试再动

✗

  • 内置 dev/analyzer/executor 仍是硬编码路径
  • 两条构建路径(硬编码 + JSON)
  • 未来还得再统一一次
02
RECOMMENDED

Unified Registry

抽一个统一的 agent_registry,内置 agent 也改成 spec 驱动,_build_agent_from_spec 成为唯一构建路径。

✓

  • 所有 agent 走同一条构建路径
  • "高扩展性"真正落地
  • 未来加 agent 零代码增量
  • 内置 prompt 仍在 .py 文件

✗

  • _build_developer_agent 要重构一次
  • _ANALYZER_READ_ONLY_TOOLS 等常量下沉到 spec
  • 需要完整的 spec 单元测试
03
FAST MVP

Incremental Minimal

只做用户自建 JSON CRUD;内置 agent 继续走原硬编码路径,短期不统一。两条路径并存一段时间。

✓

  • 改动面最小
  • 最快给终端用户看到结果
  • 风险可控

✗

  • "高扩展性"打折扣
  • 欠一笔技术债
  • 用户自建 agent 无法 delegate 到内置

关键差异:内置 agent 的构建路径

Approach 2 · Unified(推荐)
// 所有 agent 走同一条路径 registry.register( AgentSpec.from_python(DEVELOPER)) registry.register( AgentSpec.from_python(ANALYZER)) registry.register( AgentSpec.from_python(EXECUTOR)) registry.load_user_specs(store) // 运行时: spec = registry.get(primary_agent_id) agent = build_agent_from_spec( spec, modules, toolkits, memory)
Approach 3 · Minimal
// 两条路径共存 if primary_agent_id in BUILTIN: agent = _build_developer_agent(...) // 老代码完全不动 else: spec = user_store.get(primary_agent_id) agent = build_agent_from_spec(spec) // 新代码只走自建
